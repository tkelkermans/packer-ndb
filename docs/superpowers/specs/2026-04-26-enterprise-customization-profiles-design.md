# Enterprise Customization Profiles Design

## Goal

Make this repository safe and easy for enterprise customizers to extend NDB images without editing the core PostgreSQL, MongoDB, Packer, or validation roles directly.

The target user is someone who needs to add internal tooling to built images: internal CA certificates, package repositories, monitoring agents, security agents, OS hardening, backup tooling, compliance configuration, or customer-specific bootstrap logic.

The design must keep the project simple for operators:

- Use only Packer, Terraform-backed Packer plugins, Ansible, shell, and JSON/YAML files.
- Preserve the existing matrix-driven build workflow.
- Keep core database roles stable and reusable.
- Make customization auditable through manifests.
- Support committed reusable examples and private local overlays.

## Non-Goals

- Do not create a general plugin runtime or introduce a new language.
- Do not move database provisioning out of the existing Ansible roles.
- Do not store secrets in git.
- Do not make customizations required for normal PostgreSQL or MongoDB builds.
- Do not support arbitrary shell hooks before a clear Ansible profile interface exists.

## Proposed User Experience

A user chooses a normal matrix row and optionally selects a customization profile:

```bash
./build.sh \
  --ci \
  --customization-profile enterprise-example \
  --validate \
  --validate-artifact \
  --manifest \
  --ndb-version 2.10 \
  --db-type mongodb \
  --os "Rocky Linux" \
  --os-version 9.7 \
  --db-version 8.0
```

For private customer-specific logic, the user can select a local ignored profile:

```bash
./build.sh --ci --customization-profile local/customer-a ...
```

Dry runs show the selected profile, resolved profile file, enabled phases, vars files, roles, and validation roles before a live VM is created.

## Directory Layout

Add a first-class customization area:

```text
customizations/
|-- examples/
|   |-- internal-ca/
|   |   |-- roles/
|   |   `-- README.md
|   |-- monitoring-agent/
|   |   |-- roles/
|   |   `-- README.md
|   `-- os-hardening/
|       |-- roles/
|       `-- README.md
|-- profiles/
|   |-- enterprise-example.yml
|   `-- enterprise-example.vars.yml
`-- local/
    |-- .gitkeep
    `-- README.md
```

`customizations/local/` is ignored by git except for a README and optional `.gitkeep`. It is the place for customer-specific profiles, private roles, private vars, and secret-adjacent paths. The build should never require private content to exist.

## Profile Format

Profiles are YAML files. A committed example profile should be small and readable:

```yaml
name: enterprise-example
description: Install internal CA, monitoring agent, and baseline hardening.

vars_files:
  - customizations/profiles/enterprise-example.vars.yml

phases:
  pre_common:
    roles:
      - custom_internal_ca
  post_common:
    roles: []
  post_database:
    roles:
      - custom_monitoring_agent
      - custom_os_hardening
  validate:
    roles:
      - validate_custom_enterprise
```

Required fields:

- `name`: a short profile name. It must match the selected profile identifier or be clearly reported in dry-run output.
- `description`: one sentence describing the customization.
- `phases`: an object containing one or more supported phases.

Optional fields:

- `vars_files`: profile-specific variable files. Paths are relative to the repository root unless absolute.
- `extra_role_paths`: additional role roots. This allows ignored private roles under `customizations/local/...`.

Supported phases:

- `pre_common`: runs before the shared `common` role.
- `post_common`: runs after shared OS baseline work and before database provisioning.
- `post_database`: runs after database provisioning and database-specific in-guest validation.
- `validate`: runs during both in-guest validation and saved-artifact validation when validation is requested.

Unsupported phases should fail early with a clear error. Empty phases are allowed but should not be generated by the examples unless they teach the structure.

## Role Resolution

Profile roles are Ansible roles. The build should add customization role paths to Ansible's role search path without requiring users to edit `ansible.cfg`.

Role search order:

1. Existing versioned Ansible roles, such as `ansible/2.10/roles`.
2. Committed example roles under `customizations/examples/*/roles`.
3. Private local roles under `customizations/local/**/roles`.
4. Any explicit `extra_role_paths` from the selected profile.

If a role name is duplicated, Ansible's normal role resolution applies. The README should recommend unique prefixes such as `custom_` and `validate_custom_` to avoid accidental shadowing.

## Playbook Integration

The existing site playbooks should keep their database responsibilities, but gain profile hook points:

```text
pre_common custom roles
common role
post_common custom roles
database provisioning role
database validation role when --validate is set
post_database custom roles
custom validate roles when --validate is set
final image preparation role
```

Saved-artifact validation should run:

```text
database validation role
custom validate roles
```

The `validate` phase must run against the final cloned artifact as well as the temporary build VM. Otherwise a customization could pass during build but fail after image capture or first boot.

## Shell and Packer Wiring

`build.sh` owns profile selection and preflight before Packer starts.

New build options:

- `--customization-profile NAME_OR_PATH`: select a profile from `customizations/profiles`, `customizations/local`, or a direct path.
- `--no-customizations`: explicit opt-out, useful when an environment variable selects a default profile.

Optional environment variable:

- `NDB_CUSTOMIZATION_PROFILE`: default profile when the CLI flag is omitted.

Precedence:

1. `--no-customizations`
2. `--customization-profile`
3. `NDB_CUSTOMIZATION_PROFILE`
4. No customization profile

The shell layer should not parse profile YAML. Because the project already depends on `jq` but not `yq`, YAML structure validation should stay in Ansible.

Implementation detail:

- `build.sh` passes `customization_profile_file`, `customization_profile_name`, and `customization_enabled` to Ansible.
- Before Packer starts, `build.sh` runs a local Ansible preflight playbook that loads the profile file with `include_vars`.
- The preflight playbook asserts allowed phases, role list shapes, vars file existence, and explicit role path existence.
- During the image build, Ansible loads the same profile file and executes the selected role lists with `include_role` loops.

This keeps profile interpretation in Ansible, where YAML and role execution are native.

## Validation Contract

Customizations are allowed without validation, but the examples and README should strongly teach validation-first profiles.

Recommended convention:

- Install roles use `custom_` prefix.
- Validation roles use `validate_custom_` prefix.
- A production enterprise profile should include at least one `validate` role.

Examples:

- Internal CA validation checks that the certificate file exists and the OS trust store contains it.
- Monitoring agent validation checks package version, config file presence, service enabled state, and service active state.
- Hardening validation checks the specific sysctl, file permission, or service state configured by the hardening role.

If `--validate` is not set, custom validation roles do not run during the build VM phase. If `--validate-artifact` is set, custom validation roles run during artifact validation because that stage has its own validation request.

## Manifest Reporting

Manifests should record customization state:

```json
{
  "customization": {
    "enabled": true,
    "profile": "enterprise-example",
    "profile_file": "customizations/profiles/enterprise-example.yml",
    "phases": {
      "pre_common": ["custom_internal_ca"],
      "post_common": [],
      "post_database": ["custom_monitoring_agent", "custom_os_hardening"],
      "validate": ["validate_custom_enterprise"]
    },
    "validation": "passed"
  }
}
```

If no profile is selected, manifests should record `enabled: false`.

If profile parsing fails before Packer starts, the build should fail without creating a VM. If a custom validation role fails, the existing manifest failure path should mark the build failed.

## README and Onboarding

The README should gain a beginner-friendly "Customize The Image" section with this sequence:

1. Choose or copy a profile.
2. Add install roles.
3. Add validation roles.
4. Run a dry run.
5. Run one validated build.
6. Run saved-artifact validation.

Include three concrete recipes:

- Install an internal CA certificate.
- Install a monitoring or security agent.
- Add one OS hardening setting and validate it.

Each recipe should show the profile YAML, the role path, and the build command. The examples should be intentionally small, even if real enterprise tooling is more complex.

## Error Handling

Fail early before Packer when:

- The selected profile cannot be found.
- The profile cannot be loaded by Ansible.
- The profile uses an unsupported phase.
- A phase role list is not a list.
- A referenced vars file does not exist.
- A referenced explicit role path does not exist.

Fail during Ansible when:

- A customization install role fails.
- A customization validation role fails.
- A customization role expects missing required variables.

Do not hide customization failures behind warnings. A selected profile is part of the image contract.

## Testing Plan

Offline tests:

- Self-test that `build.sh --dry-run --customization-profile enterprise-example ...` reports the selected profile.
- Self-test that unknown profiles fail before Packer.
- Self-test that unsupported phases fail.
- Self-test that the local Ansible profile preflight catches malformed profile YAML before Packer.
- Self-test that README documents `--customization-profile`.
- Ansible syntax checks for both NDB versions with the example profile enabled.
- Packer validation with customization vars wired.

Live validation:

- Build one PostgreSQL row with `enterprise-example`, `--validate`, `--validate-artifact`, and `--manifest`.
- Build one MongoDB row with `enterprise-example`, `--validate`, `--validate-artifact`, and `--manifest`.
- Confirm custom install roles run in the build VM.
- Confirm custom validation roles run in both the build VM and cloned artifact.
- Confirm the manifest records the profile and validation result.

## Migration Strategy

This feature is additive. Existing commands continue to work without a profile.

Implementation should land in small slices:

1. Add profile files, example role skeletons, `.gitignore` rules, and README guidance.
2. Add shell option parsing and dry-run reporting.
3. Add the local Ansible profile preflight.
4. Add Ansible profile loading and phase dispatch.
5. Add artifact validation dispatch for custom validation roles.
6. Add manifest reporting.
7. Run offline checks and one live PostgreSQL plus one live MongoDB profile build.

## Open Decisions Resolved

- Profiles support both committed examples and ignored private overlays.
- Customization logic should be Ansible-native, not arbitrary shell hooks.
- Validation is not technically mandatory, but production examples should model it as standard practice.
- The feature should not make normal non-customized builds more complicated.
